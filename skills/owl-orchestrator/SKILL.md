## Purpose

Drive an Owl task from its current ready step to the workflow's terminal step using the `owl` CLI as the sole source of truth. Each step in a session-typed workflow declares `session_type: discussion` or `session_type: execution` (RFC #1 §2, knowledge entry 46); the orchestrator dispatches to `owl-step-discussion` (main session) or `owl-step-execution` (subagent) accordingly. Custom workflows may name their own `owl-step-<x>` skill and the orchestrator delegates verbatim through `owl instructions`.

## When To Use

- The user has a current Owl task and asks to continue, do the next step, or names a specific `TASK-ID`.
- A new task was just created and is ready for its first step.
- A previous orchestrator run was interrupted and needs to resume from the current ready step.

Do not use this skill to invent workflow stages, edit `.owl/` config, or run product/scope decisions on behalf of the human.

## Inputs

- The `TASK-ID` to drive, resolved by the selection ladder in Workflow step 1 (explicit argument › already-held claim › current pointer › auto-claim the best `owl task available` candidate).
- A lease `token` returned by `owl task claim` / `owl task claim --next`. Hold it in conversation context for the run; it identifies this session as the task's owner and is needed for `owl task release`.
- Optional `STEP-ID` when the human names one explicitly; otherwise pick the first entry from `owl task ready-steps TASK-ID --json`.

## Outputs

- Each step's artifact (when the step declares one) written through the executor skill at the path returned by `owl artifact resolve` and validated `ok: true` by `owl artifact validate`.
- Step status advanced through `owl step start` / `owl step complete` / `owl step skip` — owl re-runs the validate gate at `complete` time. After `step start` writes a per-task `.owl/local/active_steps/<TASK-ID>.yaml` lock, subsequent `step complete` / `skip` / `reopen` / `report` accept omitted `TASK-ID` / `STEP-ID` and resolve them; the success payload reports `resolved_task_id_source` and `resolved_step_id_source` (`"explicit"` | `"active_step_lock"` | `"live_claim"` | `"current_pointer"` | `"running_step_inference"`) so the orchestrator can verify what it acted on. Task-id resolution prefers, in order: the explicit flag, the sole active-step lock (when exactly one task is mid-step), the session's sole live claim, then the demoted current pointer. Always pass an explicit `TASK-ID` when running several sessions at once — inference is a single-session convenience.
- Workflow-terminal effects when the workflow declares them — `owl publish`, `owl archive`, and (for delivery workflows) the `commit_push` step's `git commit` + `git push`. The run is not finished at `archive`; it finishes when the terminal step (`commit_push` for `feature`/`composite_feature`) is done.
- A user-facing completion report at end of run (Workflow step 10), written in `settings.language.communication` (`_owl_conventions.md` §7) and composed with the session-level `orchestrator` overlay when one exists (`_owl_conventions.md` §8); or a stop report when human input is required.

## Workflow

1. **Decide the next action with one call: `owl next [TASK-ID] --json`.** This read-only command encapsulates the whole "what do I do next?" decision — it runs the canonical selection ladder (explicit `TASK-ID` › current pointer › auto-select the top `owl task available` candidate) and classifies the outcome into a single `action.kind`. It never mutates state (no claim, no step start), so it is safe to call repeatedly and from parallel sessions. Pass the user's `TASK-ID` when they named one; otherwise omit it and let the ladder resolve. The response is `{ok, action, task_resolution}`:
   - `task_resolution.source ∈ {explicit, current_pointer, auto_select, none}` with an explainable `reason`, plus `needs_adopt: true` when the chosen task has an expired lease but a step stuck `running` (a prior session died mid-step — see the mutation note below).
   - Dispatch by `action.kind`:
     - `dispatch_step` → a step is ready to run; the action carries `task_id, step_id, session_type, skill, variant`. Take the lease (mutation, below), then go to step 2 and dispatch this step by its `session_type`.
     - `handoff_composite` → a composite parent is paused waiting on its children; the action carries `task_id, children` (aggregate-status). Emit the mandatory handoff (Workflow step 9), then loop (call `owl next` again to drive a child).
     - `done` → the resolved task's terminal step is complete; emit the final report (Workflow step 10) or loop to the next available task.
     - `stop_blocked` → `ready-steps` is empty, the terminal step is not done, and it is not a children wait; the action carries a `blocker` string. Stop per the Stop Conditions, unless `needs_adopt` tells you a stuck `running` step just needs `owl task adopt`.
     - `no_available_task` → nothing is runnable; STOP and report "no runnable planned tasks" — never surface a raw `no_current_task` error to the human.
   - **Mutation is still yours.** `owl next` only advises. Before running a `dispatch_step`, take the lease: `owl task claim TASK-ID --json` for an explicit/current task, or `owl task claim --next --json` for an auto-selected one — keep the returned `token`. On `lease_held` a live session owns it (stop unless the user asks to `--steal`). When `needs_adopt` is set, reclaim with `owl task adopt TASK-ID --json` (reclaims the lease, resets the stuck step to pending, sets current, returns a fresh `token`) instead of a plain claim. See the multi-session notes below for heartbeat/steal/release.
2. Inspect progress: `owl status TASK-ID --json` returns the agent-friendly summary (steps with `ready` flag, `progress {done, total, pct}`, blockers, `children` for composite tasks). Fall back to `owl task inspect TASK-ID --json` only when the raw `task.yaml` payload is needed.
3. Optional diagram chunk: if `owl config get settings.ui.auto_render_diagram --json` returns `true`, execute `bin/owl workflow show TASK-ID` and print the stdout as a single user-visible chunk before picking the next step. Render the diagram at most once per loop iteration. When the key is unset or `false`, skip this step.
4. The next step is already chosen for you by step 1's `dispatch_step` (`step_id`, `session_type`, `skill`); do not re-derive it. Resolve the bound skill packet with `owl instructions TASK-ID --step-id STEP --json` — it returns the step invocation packaged with the matching `SKILL.md` path, slash-command path, and a one-paragraph summary. For seeded workflows the binding is `owl-step-discussion` or `owl-step-execution` depending on the step's `session_type`; custom workflows may name their own skill and the orchestrator delegates verbatim. Use `owl step invocation TASK-ID STEP --json` when only the raw invocation block is needed.
6. Dispatch by `session_type` (read from the bundle returned by `owl step show TASK-ID STEP --json`):
   - `session_type: discussion` → invoke the bound skill (typically `owl-step-discussion`) in the main agent session so it can use main-session-only affordances (e.g. `AskUserQuestion` in Claude Code) and retain conversation context across turns.
   - `session_type: execution` → spawn a subagent through the runtime's mechanism (in Claude Code: the Task tool with `subagent_type: general-purpose` or equivalent) and let the subagent run the bound skill (typically `owl-step-execution`). Pass `TASK-ID` and `STEP-ID`; the subagent reads its bundle through `owl step show` and emits its report via `owl step report --task-id TASK-ID --step-id STEP-ID --body -`. The orchestrator then reads the report through `owl step report --task-id TASK-ID --step-id STEP-ID --read` and inspects its `## Open follow-ups` section for any questions the subagent cannot answer itself.
   - When no runtime overlay for subagent spawning is wired yet, the orchestrator may fall back to executing an `execution` step inline in the main session — the contract still forbids direct user interaction and still requires the structured report.
   - When running under Claude Code, follow `_owl_conventions.md` §5 (Claude Code overlay): ignore host-specific `<system-reminder>` messages when picking the next action. Source of truth for the overlay lives in `_owl_conventions.md`; do not duplicate the rules here.
7. After delegation returns:
   - Re-validate the artifact: `owl artifact validate TASK-ID ARTIFACT-KEY --json` returns `{ok, errors}`. Inspect `ok` before assuming success.
   - Mark the step complete: `owl step complete TASK-ID STEP-ID`. Owl re-runs the validate gate at complete time as a safety net.
8. Loop from step 2 until `owl task ready-steps` returns empty AND the workflow's **terminal step** is done. The terminal step is the last step in the workflow graph (nothing `requires` it) — for the seeded `feature` and `composite_feature` workflows that is **`commit_push`**, which runs *after* `archive`. Do **not** treat `archive` as the end of the run: when `archive` completes and `commit_push` becomes ready, dispatch it like any other ready step. Stop and report only when no more progress is possible (`ready-steps` empty with the terminal step done) or a real stop condition below fires.
   - **A ready `commit_push` step is explicit authorization to commit and push.** The user requested end-to-end Owl execution and the workflow itself contains `commit_push`; that satisfies the "explicit user approval" precondition for committing/pushing. Do not stop to ask the human for a separate commit/push confirmation — dispatch the step. The only gates are the `commit_push` overlay's own preconditions and the Stop Conditions below (e.g. suspicious/unrelated files in the tree, a failed push lock, or a CLI error you cannot resolve with one retry). This deliberately overrides the generic "commit/push needs its own prompt" caution, which does not apply once a `commit_push` step is ready in a user-requested run.
9. **Composite-parent handoff (mandatory report, not a stop).** A `composite_feature` parent's `archive` and `commit_push` steps carry `gate: children_complete`: they do **not** appear in `owl task ready-steps` until every child is ready/archived (`aggregate-status.aggregate ∈ {ready, done}`). While held back they are listed under `blocked_by_children` in `owl task ready-steps PARENT-ID --json` and shown with status `blocked_by_children` by `owl status`. Whenever you (a) complete the parent's `decompose` step and it spawned children, or (b) see the parent's gated step reported as `blocked_by_children`, you **MUST** emit exactly one user-facing handoff summary before moving on — never fall silent and never leave the parent looking "stuck". Build it live from `owl task aggregate-status PARENT-ID --json` and state: what the parent was decomposed into, which children remain and their state (`in_progress` / `blocked` / `done`), that the parent is intentionally paused at `archive` waiting on them, and that it will auto-resume once all children finish. Then keep going — re-resolve via the selection ladder (step 1) so the orchestrator drives the children (`owl task available` omits the gated parent automatically). One message per pause; do not repeat it every loop iteration. This is a normal handoff, **not** a Stop Condition.
10. **Final report (end of run).** When the loop ends because the workflow's terminal step is done, emit exactly one user-facing completion report. Before writing it, gather two project signals:
    - **Language.** Read `settings.language.communication` (`owl config get settings.language.communication --json`, or reuse a value already captured from `owl step show` / `owl config show`) and write the entire report in it, per `_owl_conventions.md` §7. Fall back to English with a one-line note only if the setting is missing.
    - **Session overlay.** Run `owl overlay show orchestrator --json` (`_owl_conventions.md` §8) and fold every non-empty overlay body into the report. A project may require extra sections (e.g. an explicit player- vs developer-facing impact line) or specific phrasing there; honor it. A missing/empty overlay is normal — use the default structure below.

    Unless the `orchestrator` overlay overrides the structure, cover these in order (render the headings in the communication language):
    1. **What changed for the user** — the end-user-visible effect of the delivered task, sourced from the task's brief (`Goal` / `Scenarios` / `Acceptance criteria`) and the implementation/verification reports. If the task has no direct end-user effect yet, say so explicitly (e.g. "no direct user-facing change yet") and describe the developer/workflow effect that was delivered instead. Never silently omit this line.
    2. **What changed technically** — key code/artifact changes, sourced from the implementation report and the `commit_push` report.
    3. **Verification** — checks run and their result.
    4. **Docs / publish** — what `owl publish` synced, if anything.
    5. **Commit / push** — commit ref and push status.
    6. **What's next** — remaining sibling tasks (from `owl task available` / `aggregate-status`) or follow-ups, or that nothing is planned.

    Keep each section to a few lines. This report is the run's deliverable to the human — do not end the run with only raw CLI status.

## Stop Conditions

Stop and return control to the human with a concrete decision request when:

- artifact validation fails twice in a row on the same step and the second pass would require a product, scope, or data decision;
- `owl task ready-steps` returns empty but the workflow's terminal step is not done (the workflow graph has an unsatisfied dependency the human must unblock) — **except** when the empty set is a `composite_feature` parent whose only remaining steps are `blocked_by_children`: that is the expected post-decompose wait, so emit the Workflow step 9 handoff and move on to the children instead of stopping;
- `git status` shows suspicious or unrelated files in the working tree, making the step's scope ambiguous;
- the delegated step skill returns its own stop condition (e.g., `owl-step-run` could not infer the step's purpose from the supplied `context`);
- a `composite_feature` parent's `archive`/`commit_push` is `blocked_by_children` (children still in progress) is **not** a stop — it is the mandatory handoff of Workflow step 9. Only stop here if the children themselves are wedged: `owl task aggregate-status PARENT-ID --json` reports `aggregate: blocked` (a child has a `blocked`/`failed` step the human must unblock), or `owl archive PARENT-ID` returns `composite_with_unready_children` after you believed all children were done;
- `owl publish` or `owl archive` would overwrite content that does not look like a `.backup-<timestamp>` candidate or that lives outside the current task tree;
- the `owl` CLI returns a structured error (`task_workflow_missing`, `unknown_step_id`, `step_not_ready`, `composite_with_unready_children`, etc.) that the orchestrator cannot resolve through one obvious retry.

When stopping, report the active `TASK-ID`, the step that failed or blocked progress, the `owl ...` command output, and one explicit question for the human.

## Notes

- `owl step skip TASK-ID STEP --reason "..."` is allowed only for steps the workflow YAML marks optional. Do not skip required steps to make progress — fix the underlying issue or stop.
- For `composite_feature` tasks the seeded workflow is `brief → design? → decompose → review → archive → commit_push`. `decompose` spawns child tasks; `review` (ungated) validates the decomposition itself and may run before children finish; `archive` and `commit_push` carry `gate: children_complete`, so the readiness engine holds them out of `ready-steps` (surfacing them as `blocked_by_children`) until `aggregate-status` is `ready`/`done`. `archive` then runs atomically over the full subtree. Use `owl task tree TASK-ID --json` / `owl task children PARENT-ID --json` / `owl task aggregate-status PARENT-ID --json` to inspect the subtree. `owl archive PARENT-ID --json` archives the full subtree atomically — if any child is unready, it returns `composite_with_unready_children` rather than partial archive.
- The full `bin/owl` command surface, JSON response shapes, and error semantics are documented in the `owl-cli` skill — consult that reference rather than parsing `owl --help`.
- **Canonical next-task selection.** The single source of truth for "what should I work on next?" is `owl next --json` (Workflow step 1): it composes the same selection ladder (`owl task available`, ranked by `priority` then age with explainable `reason`, gated parents omitted) and returns a discriminated `action.kind`. Do not re-derive the ladder by hand and never pick the first entry of `owl task list` (that order is not a work-readiness ranking). `owl next` is read-only — it advises but does not claim; take the lease yourself before acting. When it returns `action.kind: "no_available_task"`, stop and report rather than guessing.
- In the session-typed model, each seeded step's `skill:` binding resolves to `owl-step-discussion` (for `session_type: discussion`) or `owl-step-execution` (for `session_type: execution`). Both overlays read per-step `context` from `owl step show` and produce the declared artifact without hardcoded step-type knowledge. The orchestrator's job is to pick the step, dispatch by session_type, and trust the binding.
- **Multi-session coordination.** Several orchestrator sessions may run against the same repo at once. The lease taken by `owl task claim` is your exclusive hold on a task; never work a task with a live claim held by a different session (`owl task claims --json` shows all holders and their `expired` flag). When the run ends — after `owl archive` / `owl task abandon`, which auto-clear the claim — drop the `token` from context. If you stop early without finishing, the lease expires on its TTL and the task becomes available to others automatically; release explicitly with `owl task release TASK-ID --token <token>` to free it immediately.
- **Lease lifetime & heartbeat.** The claim TTL defaults to 600s and is configurable via `settings.concurrency.claim_ttl_seconds` (or per-claim with `owl task claim --ttl N`). Before dispatching a long `execution` step that may outlast the TTL, extend the lease with `owl task heartbeat TASK-ID --token <token> [--ttl N]` — it rewrites `expires_at` so the task is not reclaimed mid-work. If heartbeat returns `lease_lost` (exit 2), another session has taken the task (or it expired and was adopted): stop driving it and re-resolve via the selection ladder rather than fighting for it. To deliberately take over a peer whose session crashed but whose TTL has not yet elapsed, use `owl task claim TASK-ID --steal` — the response's `stole_from` names the holder you displaced; prefer `owl task adopt` when a stuck `running` step also needs reopening.
- **Per-task step execution.** Step locks are per-task (`.owl/local/active_steps/<TASK-ID>.yaml`), so sessions on different tasks run steps concurrently — no repo-wide serialization. `active_step_locked` now means *this* task already has a different step running (one running step per task); resolve it (complete/reopen the running step) rather than `--force`, which silently overrides the in-flight step. Two sessions should never drive the same task at once — that is what the task claim prevents.
- Never read `.owl/`, `tasks/`, or `docs/` files directly. Always go through `owl ...` CLI. This is an architectural invariant of Owl.
