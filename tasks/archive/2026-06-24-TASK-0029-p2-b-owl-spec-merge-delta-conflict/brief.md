# Problem

`owl spec merge` не идемпотентен. В `delta_merger.rb` (`add`, строка ~62) `ADDED`-блок,
чьё имя требования уже присутствует в спеке, всегда возвращает `:delta_conflict`. При
повторном применении того же delta (например, после ретрая шага `merge_docs` или
повторного прогона) ранее успешно добавленное требование видится как конфликт — merge
падает, оператору приходится вручную `owl step reopen merge_docs` и разбираться. То же
для `REMOVE`/`MODIFY` несуществующей цели (`target_missing`).

# Goal

Сделать применение delta идемпотентным: повторный merge уже применённого delta —
no-op (ничего не меняет, не ошибка), при этом настоящий конфликт (то же имя требования,
но ДРУГОЕ содержимое) по-прежнему отклоняется как `delta_conflict`.

# Scenarios

### Requirement: Повторный merge применённого delta — no-op

#### Scenario: ADDED уже присутствует с тем же содержимым
- WHEN delta содержит `ADDED` требование, которое уже есть в спеке с ИДЕНТИЧНЫМ
  содержимым (повторное применение)
- THEN merge трактует это как already-applied no-op (требование не дублируется, ошибки
  нет), спек остаётся валидным

#### Scenario: повторный полный merge стабилен
- WHEN тот же delta применён дважды подряд
- THEN второй merge завершается успешно без `delta_conflict`, итоговый спек идентичен
  результату первого merge

### Requirement: Genuine-конфликт по-прежнему ошибка

#### Scenario: ADDED с тем же именем, но другим содержимым
- WHEN delta содержит `ADDED` требование с именем, которое уже есть в спеке, но с
  ДРУГИМ содержимым
- THEN merge отклоняется с `delta_conflict` (это настоящий конфликт, не идемпотентный
  повтор)

### Requirement: REMOVE/MODIFY идемпотентны

#### Scenario: REMOVE отсутствующей цели — no-op
- WHEN delta содержит `REMOVE` требования, которого уже нет в спеке (например, удалено
  предыдущим merge)
- THEN это no-op (а не `target_missing`-ошибка), merge продолжается

#### Scenario: MODIFY до идентичного содержимого — no-op
- WHEN `MODIFY` приводит требование к содержимому, которое у него уже есть
- THEN no-op, без ошибки

# Edge cases

- **Сравнение содержимого.** «Идентичность» ADDED/MODIFY определяется нормализованным
  сравнением содержимого требования (как `normalize` уже делает), чтобы незначимые
  различия форматирования не считались конфликтом. Зафиксировать критерий в plan.
- **Учёт операций.** Идемпотентные no-op'ы отражать в summary как «unchanged/already
  applied», а не как applied-изменения, чтобы счётчики были честными.
- **Граница genuine-конфликта.** Конфликт = имя совпадает И нормализованное содержимое
  различается. Только этот случай → `delta_conflict`.
- **Trace-гейт.** Если merge завязан на trace-гейт (`merge_docs`), идемпотентный повтор
  не должен ломать гейт (повторный no-op оставляет спек валидным/трассируемым).
- **Версионирование.** Изменение `lib/**` — bump `Owl::VERSION` (patch — back-compat
  устранение ложной ошибки) + CHANGELOG.

# Acceptance criteria

- [ ] `ADDED` с существующим именем + идентичным содержимым → no-op (не `delta_conflict`).
- [ ] `ADDED` с существующим именем + другим содержимым → по-прежнему `delta_conflict`.
- [ ] `REMOVE`/`MODIFY` к уже-достигнутому состоянию → no-op (не `target_missing`).
- [ ] Повторный полный `owl spec merge` того же delta идемпотентен (второй прогон
  успешен, спек стабилен).
- [ ] Summary честно отражает no-op как unchanged/already-applied.
- [ ] Тесты: идемпотентный ADDED/REMOVE/MODIFY; genuine-конфликт; двойной merge.
- [ ] `bundle exec rspec` зелёный; 100% покрытие затронутых `lib/owl/**/api.rb`;
  RuboCop net-zero.
- [ ] `Owl::VERSION` поднят + CHANGELOG.

# Out of scope

- Изменение формата delta/спека или семантики операций кроме идемпотентности.
- Условные шаги (TASK-0028). Прочие spec-команды (diff/trace/apply) — только если их
  затрагивает idempotency-фикс.
