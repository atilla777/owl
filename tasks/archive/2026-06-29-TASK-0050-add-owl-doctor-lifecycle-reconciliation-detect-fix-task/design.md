# Context

Бриф зафиксировал **узкую** задачу: команда-реконсилер `owl doctor`, которая
детектит класс дрейфа «воркфлоу терминально завершён, а `task.status` застрял
на `open`/`in_progress`» (как TASK-0041) и по `--fix` промотирует такие задачи
в `status: done`. По умолчанию — только отчёт.

**Важное уточнение к посылке брифа (по факту разведки кода).** Утверждение
брифа «ничто не флипает `status` на терминальном шаге» сегодня **неверно для
новых задач**: `Owl::Steps::Internal::TaskFinalizer` (вызывается из
`Steps::Api` при `step complete`) уже промотирует non-terminal задачу в `done`,
когда завершается её последний шаг и все шаги стали `done`/`skipped` (а для
воркфлоу с `archive` — статус уже `archived`, и его не трогают). То есть
«первопричина у источника» по сути **уже закрыта** `TaskFinalizer`. Поэтому
TASK-0041 дрейфит не из-за дыры в текущем коде, а потому что **завершён до
появления `TaskFinalizer`** (легаси). Это не отменяет ценность `owl doctor`, но
переопределяет его роль: он не «латает отсутствующий флип», а служит
**backfill / safety-net реконсилером** для (1) легаси-задач, завершённых до
`TaskFinalizer`; (2) задач, у которых статус вручную вернули в
`open`/`in_progress` после завершения шагов; (3) защиты-в-глубину, если
`TaskFinalizer` когда-либо не отработает. Решение брифа «только doctor, без
изменения жизненного цикла у источника» от этого только **усиливается**:
источник уже корректен, docter закрывает именно остаточный/легаси-слой.

Релевантная архитектура (из разведки кода):

- Команды CLI — модули `lib/owl/cli/internal/commands/*`, диспетчеризуются в
  `lib/owl/cli/api.rb`. Простые команды без подкоманд (как `next`, `status`)
  идут через таблицу `SIMPLE_COMMANDS`; команды с флагами парсят `optparse`
  (образец — `task_set_status.rb`). Команды зовут только публичные `*::Api`.
- Слой данных: `Owl::Tasks::Api` — публичная грань домена Tasks. Внутри:
  `Internal::IndexReader` (читает `tasks/index.yaml`), `read_task`/`TaskReader`
  (payload по id), `Internal::TerminalStatus.workflow_complete?(hash)` — уже
  готовый предикат «все шаги `done`/`skipped`».
- `Internal::TaskStatuses::TERMINAL = %w[archived abandoned done]` — терминальные
  task-статусы (как у `AvailabilityScanner`).
- **Ключевой реюз:** `Tasks::Api.set_status(root:, task_id:, status:)` →
  `Internal::StatusWriter` уже делает всё, что нужно для безопасной починки:
  per-task `TaskMutationLock.with_lock`, проверку `TaskSchema.settable_status?`,
  атомарную запись `task.yaml` и `IndexWriter.rebuild` (синхронизация
  `index.yaml`). `done` входит в settable-статусы.

То есть `owl doctor` целиком собирается из готовых деталей: новый код — это
тонкий read-only сканер дрейфа и тонкая CLI-обёртка; новой логики мутации не
требуется.

# Decision

**1. Где живёт сканер.** Детекция дрейфа размещается в домене **Tasks** (а не в
новом домене `Doctor`), потому что она по сути про жизненный цикл задач и
опирается на внутренние детали Tasks (`IndexReader`, `read_task`,
`TerminalStatus`). Правило слоёв запрещает кросс-доменный доступ к `Internal::*`,
поэтому выносить сканер в отдельный домен означало бы либо протечку internals,
либо лишнюю публичную грань. Узкий охват из брифа отдельный домен не оправдывает.

- Новый внутренний модуль `Owl::Tasks::Internal::DriftScanner` (read-only):
  enumerate неархивные записи индекса → для каждой `read_task` → дрейф, если
  `TerminalStatus.workflow_complete?(payload) == true` И
  `payload['status'] ∈ {open, in_progress}`. Возвращает список
  `{task_id, status, workflow, terminal_step_id, suggested_status: 'done'}`.
- Новая публичная грань `Owl::Tasks::Api.lifecycle_drift(root:)` → делегирует в
  `DriftScanner`, возвращает `Result.ok(drifted: [...])`. **Read-only** — не
  мутирует ни `task.yaml`, ни `index.yaml`.

**2. Предикат «терминально завершён».** Используем `workflow_complete?` (все
шаги `done`/`skipped`), а не «единственный терминальный шаг `done`». Для
корректного DAG это эквивалентно (терминальный шаг `requires` всю цепочку), но
переиспользует уже оттестированный код и корректно учитывает `skipped`-шаги
(условные `when:`) и gated-терминалы композитных родителей (их `archive`/
`commit_push` не `done`, пока дети не готовы → родитель в середине потока не
помечается дрейфом). `terminal_step_id` для отчёта берём как шаг, который никто
не `requires`.

**3. Починка реюзает `set_status`.** `--fix` не вводит новый путь мутации: для
каждой дрейфующей задачи команда вызывает существующий
`Tasks::Api.set_status(status: 'done')`. Это бесплатно даёт per-task лок,
schema-валидацию и rebuild индекса; согласованность `task.yaml ↔ index.yaml`
гарантирована тем же кодом, что и `owl task set-status`. Идемпотентность — из
определения дрейфа: после установки `done` задача выходит из множества
`{open, in_progress}` и в следующий прогон не попадает.

**4. CLI.** Новая top-level команда `owl doctor [--fix] [--json] [--root PATH]`,
зарегистрированная как `SIMPLE_COMMAND` → `Internal::Commands::Doctor`. Без
`--fix`: зовёт `lifecycle_drift`, печатает `{ok, drifted:[...], fixed:[]}`,
exit 0. С `--fix`: после скана вызывает `set_status('done')` на каждую
дрейфующую задачу, печатает `{ok, drifted:[...], fixed:[{task_id, from, to}]}`.
JSON по умолчанию (как весь `bin/owl`). Регистрируется в `HelpText`.

**5. Только реконсилер.** Жизненный цикл у источника не меняем — для новых задач
его уже обеспечивает `TaskFinalizer` (см. Context), а doctor покрывает
легаси/ручные ре-опены. `done`, а не `archived`:
doctor не перемещает файлы и не трогает архив; перенос в архивное хранилище
остаётся за `owl archive`.

# Alternatives

- **A1. Авто-выставление `status=done` на терминальном шаге (фикс у источника).**
  Убрал бы дрейф для новых задач — **и это уже реализовано** в
  `TaskFinalizer` (см. Context). Поэтому отдельный source-side фикс в этой
  задаче не нужен и не делается (решение брифа «только doctor» совпало с уже
  существующим положением дел); `owl doctor` остаётся backfill-слоем для легаси
  и ручных ре-опенов.
- **A2. Отдельный домен `Owl::Doctor` вместо метода в `Tasks::Api`.** Дал бы
  место для роста (общий health-check), но потребовал бы доступа к
  `Tasks::Internal::*` (протечка слоёв) либо дублирующей публичной грани ради
  узкой фичи. Выбран метод в `Tasks::Api`; если чек-набор расширится —
  выделение `Doctor`-домена станет оправданным рефактором.
- **A3. Новый bespoke-путь мутации для `--fix`.** Свой writer для статуса.
  Отклонено: дублировал бы лок/schema/index-rebuild, уже идеально решённые в
  `StatusWriter`. Реюз `set_status` — меньше кода и нулевой риск рассинхрона.
- **A4. Предикат «единственный терминальный шаг `done`» вместо
  `workflow_complete?`.** Чуть точнее формулировке брифа, но требует находить
  терминальный шаг и игнорирует семантику `skipped`. Выбран `workflow_complete?`
  как оттестированный и эквивалентный для корректных DAG.

# Risks

- **Нарушение ортогональности статуса.** `--fix` выводит статус из шагов, тогда
  как схема говорит «never derived from steps». Митигировано: по умолчанию
  только отчёт; `--fix` — явный opt-in; промотируются лишь `open`/`in_progress`
  (явные `blocked`/`on_hold` неприкосновенны); документируется как реконсилер
  постфактум, а не автоматический жизненный цикл.
- **Гонка scan→fix в параллельных сессиях.** Между сканом и `set_status` другая
  сессия могла сменить статус. `StatusWriter` перечитывает payload под локом и
  валидирует схему; худший случай — `done` ставится на задачу, которая уже
  `workflow_complete?`, что всё равно корректно. Опционально: в `--fix`-цикле
  перепроверять дрейф перед каждым `set_status`.
- **Частичный прогон `--fix` по многим задачам.** Каждая задача чинится
  независимо и атомарно; сбой на одной не откатывает другие и оставляет
  остаток на следующий (идемпотентный) прогон.
- **Ложные срабатывания на композитных родителях в середине потока.**
  `workflow_complete?` требует ВСЕ шаги `done`/`skipped`; gated `archive`/
  `commit_push` не `done`, пока дети не готовы → mid-flow родитель не
  помечается. Риск закрыт выбором предиката (A4).
- **`done` ≠ архив.** После `doctor --fix` задача `done`, но физически не
  заархивирована; для переноса нужен `owl archive`. Документируется, чтобы
  `done`-без-архива не воспринимался как баг.

# API

Публичная поверхность, публикуемая в `docs/` при мерже:

- **CLI**
  - `owl doctor [--json] [--root PATH]` — read-only скан дрейфа. Вывод:
    `{ "ok": true, "drifted": [ { "task_id", "status", "workflow",
    "terminal_step_id", "suggested_status": "done" } ], "fixed": [] }`. Не
    мутирует состояние. Exit 0.
  - `owl doctor --fix [--json] [--root PATH]` — реконсиляция. Для каждой
    дрейфующей задачи `set_status('done')` под per-task локом. Вывод:
    `{ "ok": true, "drifted": [...], "fixed": [ { "task_id", "from", "to":
    "done" } ] }`. Идемпотентна. Exit 0.
- **Ruby (домен Tasks)**
  - `Owl::Tasks::Api.lifecycle_drift(root:) -> Result.ok(drifted: Array<Hash>)`
    — read-only детекция; единственная новая публичная грань (требует 100%
    покрытия как `lib/owl/tasks/api.rb`).
  - Реюз без изменения сигнатуры: `Owl::Tasks::Api.set_status(root:, task_id:,
    status:)` — путь починки.
- **Внутреннее (без публичного контракта):**
  `Owl::Tasks::Internal::DriftScanner`, `Owl::Cli::Internal::Commands::Doctor`.

Совместимость: только аддитивно (новая команда + новый метод Api + новые JSON-
ключи в выводе новой команды). Существующие контракты не меняются. Бамп —
**minor** (новая фича) + запись в `CHANGELOG.md`.
