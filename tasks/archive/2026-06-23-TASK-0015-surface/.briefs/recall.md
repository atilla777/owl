# Brief — Кросс-задачная память: surface похожих архивных задач/решений в бриф

## Problem

Когда автор (человек или агент) пишет `brief` для новой задачи, он не видит,
что похожая проблема уже решалась в одной из **архивных** задач. Накопленный
закрытый опыт (`tasks/archive/**`, доступный через `owl archive list|show|read`)
лежит мёртвым грузом: его нельзя дёшево «вспомнить» в момент постановки задачи.
Последствия:

- дублирование решений и повторное прохождение уже пройденных развилок;
- расхождение подходов между задачами, которые стоило бы сделать одинаково;
- потеря связи «новая задача ↔ её предшественники», которую человеку пришлось
  бы восстанавливать вручную, перебирая архив.

Сейчас в Owl нет команды, которая по короткому запросу (теме новой задачи)
вернула бы ранжированный список релевантных архивных задач. `owl archive list`
лишь перечисляет архив целиком, без поиска и ранжирования.

## Goal

Дать Owl **кросс-задачную память**: дешёвый, детерминированный, не требующий
внешних зависимостей механизм, который по текстовому запросу возвращает
ранжированный список похожих **архивных** задач, и сделать его доступным автору
брифа.

Конкретно:

1. Новая read-only CLI-команда `owl recall <query> [--json] [--limit N]`.
2. Лексическое сопоставление (token-overlap / tf-idf на чистом Ruby) запроса
   с корпусом = **архивные задачи** (их `title` + секции `Problem`/`Goal`
   из `brief`).
3. Результат — ранжированный список совпадений: `task_id`, `title`, `score`,
   1-строчный `snippet` совпавшего текста.
4. На шаге `brief` (через `owl-step-discussion` / оркестратор) этот результат
   показывается автору как «Похожие архивные задачи», чтобы переиспользовать
   опыт до написания брифа.

Корпус читается **только** через `bin/owl` (archive-роль: `owl archive
list|read`), без прямого чтения файлов `tasks/`/`docs/` — это инвариант
проекта (источник-истины и слой Backend/Internal/Api,
`docs/agents/27_Owl_Ruby_code_architecture.md`).

### Не входит в объём (Non-goals)

- Семантический поиск через embeddings / внешние ML-API — явно отвергнут
  (выбран лексический путь без сетевых зависимостей).
- Индексация активных/в-работе задач — корпус ограничен **архивом**
  (исключает self-match и шум незавершённого).
- Автоматическая правка брифа на основе найденного — `recall` только
  *подсказывает*, решение о переиспользовании за автором.
- Персистентный индекс/кэш на диске — на текущем масштабе (десятки архивных
  задач) допустим прямой проход по корпусу при каждом вызове; кэш — возможная
  будущая оптимизация, не требование.

## Scenarios

### Requirement: recall-команда

The system SHALL предоставлять read-only CLI-команду `owl recall <query>`,
возвращающую ранжированный по релевантности список похожих архивных задач.

#### Scenario: запрос находит похожие архивные задачи
- WHEN пользователь выполняет `owl recall "<тема новой задачи>" --json`
- THEN команда возвращает `{ok: true, matches: [...]}`, где каждый элемент —
  `{task_id, title, score, snippet}`
- AND элементы отсортированы по убыванию `score`
- AND результат не мутирует никакое состояние репозитория (команда read-only)
- TEST: spec/owl/recall/api_spec.rb

#### Scenario: ограничение числа результатов
- WHEN пользователь выполняет `owl recall "<запрос>" --limit 5`
- THEN команда возвращает не более 5 совпадений (топ по `score`)
- TEST: spec/owl/recall/api_spec.rb

#### Scenario: нет релевантных совпадений
- WHEN запрос не пересекается по значимым токенам ни с одной архивной задачей
- THEN команда возвращает `{ok: true, matches: []}` и нулевой exit-код
- AND не падает с ошибкой
- TEST: spec/owl/recall/api_spec.rb

### Requirement: лексическое ранжирование без внешних зависимостей

The system SHALL вычислять релевантность лексически (пересечение
токенов / tf-idf) на чистом Ruby, без сетевых вызовов и внешних gem-ов.

#### Scenario: ранжирование по совпадению термов
- WHEN запрос делит значимые токены с `title` или секциями `Problem`/`Goal`
  брифа архивной задачи
- THEN такая задача получает `score` > 0, растущий с числом/весом совпавших
  термов, и попадает выше менее релевантных
- TEST: spec/owl/recall/ranking_spec.rb

#### Scenario: отсутствие сетевых/внешних зависимостей
- WHEN команда выполняется в офлайне без каких-либо ключей/API
- THEN она отрабатывает корректно и детерминированно (тот же запрос на том же
  корпусе → тот же порядок)
- TEST: spec/owl/recall/ranking_spec.rb

### Requirement: корпус = архив, доступ через bin/owl

The system SHALL строить корпус поиска только из архивных задач, читая их
через archive-роль CLI, и MUST NOT обращаться к файлам `tasks/`/`docs/`
напрямую в обход слоя Backend/Internal.

#### Scenario: в корпус входят только архивные задачи
- WHEN в репозитории есть и активные (`tasks/TASK-XXXX`), и архивные
  (`tasks/archive/**`) задачи
- THEN `owl recall` сопоставляет запрос только с архивными задачами
- AND активные задачи в результат не попадают
- TEST: spec/owl/recall/corpus_spec.rb

#### Scenario: доступ к корпусу инкапсулирован
- WHEN retrieval читает `title`/`brief` архивных задач
- THEN он делает это через тот же путь данных, что и `owl archive list|read`
  (Backend/Internal), а не через прямые `File.read` в Api/CLI-слое
- TEST: spec/owl/recall/corpus_spec.rb

### Requirement: surface на шаге brief

The system SHALL делать результат `recall` доступным автору на шаге `brief`
(через `owl-step-discussion`/оркестратор), чтобы похожие архивные задачи
были видны до написания брифа.

#### Scenario: автор брифа видит похожие архивные задачи
- WHEN запускается шаг `brief` для новой задачи с непустым заголовком
- THEN автору показывается секция «Похожие архивные задачи» с топ-совпадениями
  (`task_id`, `title`, `snippet`), полученными из `owl recall`
- AND если совпадений нет — это явно сообщается, шаг не блокируется
- TEST: spec/owl/recall/brief_surface_spec.rb

## Edge cases

- **Пустой архив.** Нет архивных задач → `matches: []`, exit 0, без ошибок.
- **Пустой/тривиальный запрос** (пустая строка или только стоп-слова) →
  `matches: []` (или валидационная ошибка с понятным сообщением); не падать.
- **Архивная задача без `brief`** (или без секций Problem/Goal) → в корпус
  входит хотя бы её `title`; отсутствие брифа не роняет команду.
- **Кириллица/смешанные алфавиты.** Токенизация устойчива к русскому тексту
  (заголовки и брифы проекта на русском) — нормализация регистра/юникода.
- **Self-match при будущем surface.** Если/когда текущая задача уже в архиве,
  она не должна доминировать в собственном recall (на шаге brief текущая задача
  ещё не в архиве, так что для основного сценария проблема не возникает —
  зафиксировать как известное ограничение).
- **Дубль-сниппет.** Сниппет — короткий (1 строка) фрагмент совпавшего текста,
  усечённый по длине; без переносов/markdown-ломки JSON.
- **Стабильность сортировки.** При равных `score` порядок детерминирован
  (вторичный ключ — например, `task_id`), чтобы вывод был воспроизводим.

## Acceptance criteria

1. `owl recall "<query>" --json` возвращает `{ok, matches:[{task_id, title,
   score, snippet}]}`, отсортированные по убыванию `score`; `--limit N`
   усекает до N; пустой результат даёт `matches: []` и exit 0.
2. Ранжирование лексическое (token-overlap/tf-idf), чистый Ruby, без сетевых
   вызовов и новых внешних gem-ов; детерминированно при равных входных данных.
3. Корпус — только архивные задачи (`title` + `Problem`/`Goal` из brief),
   читаемые через archive-роль; активные задачи не индексируются; прямых
   `File.read` в обход Backend/Internal/Api нет (соответствие
   `docs/agents/27_Owl_Ruby_code_architecture.md`).
4. Публичный API-слой команды (`lib/owl/**/api.rb`-эквивалент для recall)
   покрыт тестами на 100% строк (соответствие
   `docs/agents/30_Owl_Ruby_testing_RSpec_and_public_API_coverage.md`);
   названы spec-файлы (api/ranking/corpus/brief_surface).
5. Результат `recall` доступен на шаге `brief` и показывается автору
   («Похожие архивные задачи»); отсутствие совпадений сообщается и не
   блокирует шаг.
6. Изменение бампит `Owl::VERSION` (новая команда = minor) и добавляет запись
   в `CHANGELOG.md` в том же коммите (Конституция §7.1); JSON-контракт
   `owl recall` задокументирован.
7. Edge cases выше либо покрыты поведением, либо явно зафиксированы как
   известные ограничения.

> Декомпозиция (шаг `decompose`) ожидаемо разрежет это на неперекрывающиеся
> дочерние задачи — ориентировочно: (a) лексический retrieval-движок
> (токенизация + скоринг + чтение корпуса через archive-роль),
> (b) команда `owl recall` + Api + JSON-контракт + специи,
> (c) surface на шаге `brief`. Точные границы детей определяет `decompose`.
