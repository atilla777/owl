# Problem

Многосессионный режим Owl защищает задачи per-task lease и per-task step-локом, а
пуши — repo-scoped git-локом. Но запись общего файла `tasks/index.yaml` ничем не
сериализована. `index_rebuilder` пишет файл атомарно (write+rename), что спасает от
**битого** файла, но не от **потерянного обновления**: если сессия A и сессия B
одновременно изменяют ростер (create/archive/delete/rebuild), обе читают старое
состояние, обе пишут — последний rename побеждает, чужое изменение теряется.

Это реальный риск ровно в том сценарии, который Owl поощряет (параллельные
оркестраторы), и он уже всплыл косвенно при текущей доставке: backlog-задачи и индекс
приходится разводить вручную.

# Goal

Сделать все мутации `tasks/index.yaml` взаимно исключающими через существующий
механизм `Owl::Locks` (repo-scoped именованный лок), чтобы конкурентные
create/archive/delete/rebuild не теряли записей и индекс всегда отражал реальный набор
`task.yaml` на диске.

# Scenarios

### Requirement: Конкурентные мутации индекса не теряют записей

#### Scenario: два параллельных create не затирают друг друга
- WHEN две сессии одновременно выполняют `owl task create` (каждая добавляет свою
  задачу в ростер)
- THEN после завершения обеих `tasks/index.yaml` содержит ОБЕ задачи, ни одна запись
  не потеряна

#### Scenario: create и archive из разных сессий сериализуются
- WHEN сессия A архивирует задачу (удаляет её из активного ростера), а сессия B в тот
  же момент создаёт новую задачу
- THEN итоговый индекс отражает оба изменения (архивированная отсутствует, новая
  присутствует)

### Requirement: Лок не вносит дедлоков и самоблокировок

#### Scenario: одиночная сессия не блокирует сама себя
- WHEN одна сессия выполняет обычную цепочку (create → ... → archive)
- THEN каждая операция захватывает и освобождает индекс-лок без зависаний; лок имеет
  TTL и освобождается при сбое сессии (как прочие Owl-локи)

# Edge cases

- **TTL и отказ.** Индекс-лок должен иметь TTL и авто-освобождаться, чтобы упавшая
  сессия не заклинила ростер навсегда (консистентно с lease/step-локами).
- **Вложенность с другими локами.** Операции, уже держащие per-task lease/step-лок,
  не должны конфликтовать/дедлочиться с индекс-локом (определить порядок захвата).
- **Атомарность сохраняется.** Существующий write+rename остаётся; лок добавляется
  ВОКРУГ read-modify-write, а не вместо атомарной записи.
- **Идемпотентность rebuild.** `owl task index rebuild` под локом даёт тот же
  результат, что и сейчас, просто безопасно при гонке.
- **Версионирование.** Изменение `lib/**` — consumer-visible: bump `Owl::VERSION`
  (patch — back-compat hardening) + `CHANGELOG.md`.

# Acceptance criteria

- [ ] Все пути записи `tasks/index.yaml` (create, archive, delete, set-priority,
  index rebuild и любые другие мутаторы ростера) выполняют read-modify-write под
  единым именованным `Owl::Locks` локом (напр. `index`).
- [ ] Лок имеет TTL и освобождается в `ensure`/при сбое; нет самоблокировки в обычной
  одиночной цепочке.
- [ ] Регрессионный тест на гонку: смоделировать конкурентные мутации (или прямой
  тест на сериализацию через лок-API) — обе записи сохраняются.
- [ ] `bundle exec rspec` зелёный; 100% покрытие затронутых `lib/owl/**/api.rb`;
  RuboCop net-zero на трогаемых файлах.
- [ ] `Owl::VERSION` поднят (patch) + запись в `CHANGELOG.md`.

# Out of scope

- Полный переход на SQLite-бэкенд (P3) — здесь только лок над текущим FS-индексом.
- Безопасный scoped-staging для `owl commit-push` (отдельная находка/задача).
