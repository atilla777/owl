# Context

Задача трогает три независимых поверхности сборки, у каждой — свой выбор
подхода. Только первая (SimpleCov-гейт) содержит нетривиальное архитектурное
решение; две другие — преимущественно исполнение по уже принятым в брифе
решениям, но фиксируются здесь, чтобы план и реализация работали из одной
записи.

Затрагиваемые точки:
- `spec/spec_helper.rb` — старт SimpleCov и хук `at_exit` с гейтом 100% на
  публичных API-файлах.
- `.rubocop.yml` — конфигурация линтера (77 нарушений: 39 авто, ~38 ручных).
- `.github/workflows/` — отсутствует; remote GitHub `atilla777/owl`.
- `lib/**/*.rb` — отдельные файлы, где RuboCop требует ручной правки кода
  (LineLength, Metrics, UnusedMethodArgument и т.п.).

# Decision

**1. SimpleCov-гейт активен только на полном прогоне — через сравнение
`files_to_run`.** В хуке `SimpleCov.at_exit` перед проверкой покрытия
сравниваем множество файлов, которые RSpec реально запускал
(`RSpec.configuration.files_to_run`, абсолютные пути), с полным набором,
обнаруженным `Dir.glob('spec/**/*_spec.rb')` (раскрытым в абсолютные пути).
Если множества совпадают — это полный прогон, гейт работает как раньше
(печать нарушителей + `exit 1`). Если запущено подмножество — гейт молча
выходит (`next`), не трогая код выхода процесса. Логика детекции выносится в
маленький приватный helper в `spec_helper.rb`, покрытый прямым unit-тестом на
само сравнение множеств.

**2. RuboCop приводится к 0 нарушениям комбинированно.** Сначала
`bundle exec rubocop -a` (safe autocorrect, 39 шт.). Остаток разбивается:
нарушения в `lib/` чинятся кодом (перенос строк, упрощение методов, удаление
неиспользуемых аргументов); spec-копы (`RSpec/LeakyLocalVariable`,
`RSpec/LeakyConstantDeclaration`, `RSpec/ExampleLength`,
`Lint/ConstantDefinitionInBlock` в спеках) гасятся точечным послаблением
`.rubocop.yml` для `spec/**` с коротким комментарием-обоснованием. Цель —
`bundle exec rubocop` exit 0 без `.rubocop_todo.yml`.

**3. CI — один workflow `.github/workflows/ci.yml`, один job, два
блокирующих шага.** Триггеры `push` и `pull_request`. Шаги:
`actions/checkout` → `ruby/setup-ruby` (Ruby 3.3, `bundler-cache: true`) →
`bundle exec rspec` (полный прогон → гейт активен) → `bundle exec rubocop`.
Любой ненулевой код валит workflow. Минимальные права (`contents: read`).

# Alternatives

**Гейт — детекция полного прогона:**
- *Авто-детект через `files_to_run` (выбрано).* Прозрачно, без ручных флагов,
  ровно отражает требование брифа; покрывается unit-тестом.
- *ENV-флаг `COVERAGE_GATE=1` (отклонено пользователем).* Явно и просто, но
  локально про флаг надо помнить, и полный локальный прогон по умолчанию
  переставал бы проверять покрытие — против духа гейта.
- *`SimpleCov.minimum_coverage_by_file 100` (встроенное).* Применяется ко
  всем track_files, а не только к `api.rb`/`result.rb`, и так же сорвётся на
  частичном прогоне — не решает исходную проблему, меняет семантику гейта.

**RuboCop — путь к зелёному:**
- *Полностью зелёный код + spec-послабления (выбрано).* Чистый базис для
  блокирующего линт-гейта, без техдолга в виде todo-файла.
- *`.rubocop_todo.yml` baseline.* Быстрее, но замораживает текущие нарушения
  как «разрешённые» — линт-гейт тогда не ловит их при правках, маскирует долг.

**CI — структура:**
- *Один job, два шага (выбрано).* Минимально, быстро, ровно по брифу.
- *Раздельные jobs rspec и rubocop (матрица).* Чуть нагляднее статусы, но
  дублирует bundle install и усложняет минимальный workflow без выгоды на
  этом размере проекта.

# Risks

- **Хрупкость детектора полного прогона.** Если glob `spec/**/*_spec.rb`
  разойдётся с тем, что RSpec считает набором по умолчанию (нестандартный
  `.rspec`, кастомный pattern), полный прогон может ошибочно считаться
  частичным и гейт замолчит. Митигация: glob совпадает с дефолтным паттерном
  RSpec; unit-тест на helper; CI всегда гонит полный набор, так что регрессия
  покрытия всё равно поймается на push.
- **Регрессия покрытия при ручной правке `api.rb`/`result.rb` под RuboCop.**
  Митигация: полный прогон с активным гейтом локально и в CI — красный при
  падении ниже 100%.
- **spec-послабления расширены слишком широко.** Митигация: послабления
  ограничены `spec/**` и сопровождены комментарием; копы в `lib/` не глушатся.
- **`bundler-cache`/setup-ruby в CI.** При проблемах с кэшем job не должен
  падать по инфраструктуре — кэш опционален, базовый путь `bundle install`
  обязан проходить на Ruby 3.3.

# API

Публичной поверхности `bin/owl` (CLI-команды, JSON-контракты, схемы,
workflow/artifact-определения) эта задача **не меняет** — публиковать в
`docs/` через `publishes:` нечего. Затрагиваются только наблюдаемые
контракты разработческого тулинга:

- **Контракт coverage-гейта (`spec/spec_helper.rb`).** Гейт 100% покрытия
  публичного API (`lib/owl/**/{api,result}.rb`) срабатывает и может дать
  `exit 1` **только** на полном прогоне suite (`files_to_run` == полный glob
  `spec/**/*_spec.rb`). На любом частичном прогоне гейт бесшумен, код выхода
  определяется только результатом тестов.
- **Контракт CI (`.github/workflows/ci.yml`).** На `push` и `pull_request`
  выполняется `bundle install` → полный `rspec` → `rubocop`; оба шага
  блокирующие, ненулевой код любого валит workflow.
- **Контракт RuboCop (`.rubocop.yml`).** `bundle exec rubocop` по репозиторию
  завершается с кодом 0; послабления существуют только для `spec/**`.
